Content Module

Overview

The Content module manages WordPress WP_Post objects — posts, pages, and any registered custom post type. It wraps WP_Query, wp_insert_post(), wp_update_post(), and wp_delete_post() behind a consistent, permission-aware interface.

All write and delete abilities are pro-gated by default. Read abilities are available on the free tier. The permissions filter wp_abilities_suite_permissions controls access per user role.

Block markup notecontent/get returns raw Gutenberg block markup. If you pass this directly to content/update, the block structure is preserved. Use the Blocks module (blocks/parse, blocks/insert) to surgically modify content without touching the full post.

Common Workflows

Publish a blog post

1. content/create        → post_status: "draft", post_type: "post"
2. taxonomies/assign     → attach category + tag term IDs
3. media/upload          → upload featured image, get attachment ID
4. meta/update-post-meta → set _thumbnail_id to attachment ID
5. content/update        → post_status: "publish"

Survey site content structure

1. content/discover-types     → see all post types
2. content/get-site-map       → hierarchical page tree
3. content/list-structure     → posts list without content (fast)
4. taxonomies/list-terms      → categories and tags in use

Surgical content edit

1. blocks/parse              → get block array from post_content
2. blocks/find-in-post       → locate target block by type/attribute
3. blocks/update-attributes  → change only what needs changing
4. content/update            → write modified block markup back

Known Gotchas

Block markup preservationIf you pass a plain string (not block markup) to content/update as post_content, WordPress will save it as a Classic block. Always use blocks/serialize if you're constructing content programmatically.
post_author not yet supportedcontent/create and content/update do not accept a post_author parameter. Posts default to the bridge user. Known gap — GitHub issue filed, queued for next sprint.
search-replace on large sitesAlways use dry_run: true first. On sites with thousands of posts, consider using post_ids to batch the operation.
